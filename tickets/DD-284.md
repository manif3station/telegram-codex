# DD-284

## Summary

Move the always-on Telegram polling path under a DD collector so `dashboard telegram-codex.start` creates or heals one collector per workspace session and routes two-way Telegram replies through `dashboard telegram-codex.check-messages`.

## Method

TDD

## Acceptance Criteria

- `dashboard telegram-codex.start` ensures exactly one `telegram-codex-<session-id>` collector exists in `~/.developer-dashboard/config/config.json`
- duplicate collector entries for that session are removed automatically
- the collector uses:
  - `interval: 5`
  - `rotation.lines: 100`
  - `cwd` fixed to the workspace where start ran
  - `command: dashboard telegram-codex.check-messages`
  - `mode: singleton`
- `dashboard telegram-codex.start` persists the active Codex reply target in `~/.telegram-codex/<session-id>/codex.session`
- `dashboard telegram-codex.start` runs `dashboard restart collector telegram-codex-<session-id>`
- `dashboard telegram-codex.check-messages` remains the long-running polling loop that receives text and multimedia update metadata and routes Telegram replies back through the active Codex session
- docs explain the collector-owned runtime instead of the old standalone listener path
- Docker tests pass
- coverage reaches `100%`
- `cover_db` is cleaned
- `README.md`, `docs/`, `Changes`, and `tickets/` are updated
- release commit and push complete from the `telegram-codex` repo
